### Q10: How did you manage state in your applications without Redux?

**Answer:**
I used **RxJS Observables and BehaviorSubject** as the state management pattern:

**The Pattern:**
```typescript
// Service provides single source of truth
class TaskService {
  private taskSubject = new BehaviorSubject<Task[]>([]);
  tasks$ = this.taskSubject.asObservable(); // Expose as observable
  
  // Get current value synchronously
  getTasks(): Task[] {
    return this.taskSubject.value;
  }
  
  // Update state immutably
  addTask(task: Task): void {
    const current = this.taskSubject.value;
    this.taskSubject.next([...current, task]);
  }
}

// Component subscribes to changes
constructor(private taskService: TaskService) {}
tasks$ = this.taskService.tasks$; // Observable in component

// Template auto-subscribes with async pipe
<task-item *ngFor="let task of tasks$ | async"></task-item>
```

**Why This Works:**
- **Single Source of Truth** - All data comes from BehaviorSubject
- **Immutable Updates** - Creating new arrays prevents bugs
- **Auto-Unsubscribe** - Async pipe handles subscription cleanup
- **Type-Safe** - TypeScript ensures data integrity
- **Reactive** - Changes automatically propagate to UI

**When to Use Redux:**
If the application had 5000+ components or deeply nested state, Redux would add structure and time-travel debugging.

## 4. Challenges & Problem-Solving Questions

### Q13: Tell us about the biggest technical challenge you faced and how you solved it.

**Challenge: RxJS Observable Race Conditions**

**The Problem:**
In the Task Manager app, when users did multiple operations quickly (add, delete, complete), tasks would:
- Appear then disappear
- Show incorrect completion status
- Lose data silently

**Root Cause:**
Multiple async pipes creating separate subscriptions, each with its own timing, leading to data inconsistency.

**Debugging Process:**
1. Reproduced issue consistently (rapid clicking)
2. Added console logs to track data flow
3. Realized multiple subscriptions were out of sync
4. Researched RxJS patterns

**The Solution:**
Implemented single BehaviorSubject as centralized store:
```typescript
// Before: Multiple subscriptions
{{ (tasks$ | async)?.length }}        // Subscription 1
<div *ngIf="tasks$ | async as tasks"> // Subscription 2
  <task *ngFor="let t of tasks">      // Subscription 3 per item

// After: Single subscription with one source of truth
<ng-container *ngIf="(tasks$ | async) as tasks">
  {{ tasks.length }}
  <div>
    <task *ngFor="let t of tasks">
</ng-container>
```

**Result:**
- Data consistency restored
- Subscriptions properly managed
- No more flickering or data loss

**Learning:**
This taught me:
- How reactive patterns need careful orchestration
- Importance of single source of truth
- Testing async code thoroughly
- When observables shine vs. when they cause problems

### Q14: Describe a Git problem you faced and how you resolved it.

**Challenge: Git History Divergence**

**The Problem:**
After someone did a force push on the remote, my local branch fell behind:
```
error: failed to push some refs 
! [rejected] main -> main (non-fast-forward)
```

**Why It Happened:**
Remote history changed (force push), but my local commits weren't part of the new history. Pushing would create a conflict.

**Solution:**
```bash
# 1. Fetch latest from remote
git fetch origin

# 2. Rebase my commits on top of new history
git rebase origin/main

# 3. If conflicts, resolve manually then:
git rebase --continue

# 4. Push with confidence
git push origin main
```

**Key Understanding:**
- Git DAG (Directed Acyclic Graph) model
- Fast-forward vs. three-way merges
- Why force push is dangerous in teams
- How to recover from divergent histories

**Prevention Learned:**
- Keep branches synchronized with `git pull --rebase`
- Communicate before force pushing
- Use pull requests instead of direct pushes
- Understand implications of rewriting history

## 6. Best Practices & Professional Development

### Q19: What are your coding best practices?

**Answer:**
Principles I follow consistently:

**Readability First**
```typescript
// ❌ Clever but hard to understand
const tasks = t.filter(x => x.s === 'c').map(x => ({...x, d: new Date()}));

// ✅ Clear intent
const completedTasks = tasks
  .filter(task => task.status === 'completed')
  .map(task => ({...task, completedDate: new Date()}));
```

**DRY (Don't Repeat Yourself)**
```typescript
// ❌ Duplicated validation logic
if (!name || name.length < 3) showError();
if (!email || email.length < 5) showError();

// ✅ Reusable validator
const isValidField = (value: string, minLength: number) => 
  value && value.length >= minLength;
```

**SOLID Principles**
- **S**ingle Responsibility - One class, one reason to change
- **O**pen/Closed - Open for extension, closed for modification
- **L**iskov Substitution - Derived classes maintain base contract
- **I**nterface Segregation - Small, focused interfaces
- **D**ependency Inversion - Depend on abstractions, not concretions

**Testing**
- Unit test critical business logic
- Component tests for user interactions
- E2E tests for critical user flows
- Aim for 80%+ coverage on core features

**Documentation**
- Code comments explain "why", not "what"
- README explains how to run/build
- API documentation with examples
- Architecture decisions documented

## 7. Rapid-Fire Questions

Quick answers that show breadth of knowledge:

### Q21: What's your go-to debugging technique?
**Answer:** Console logging strategically placed + Chrome DevTools + rubber duck debugging (explaining to colleague).

### Q22: How do you optimize Angular application performance?
**Answer:** ChangeDetectionStrategy.OnPush, lazy loading, code splitting, async pipe, OnDestroy cleanup, profiling with Lighthouse.

### Q23: What's the difference between let, const, and var in TypeScript?
**Answer:** var (function-scoped, hoisted, avoid), let (block-scoped, mutable, preferred), const (block-scoped, immutable, use by default).

### Q24: Explain async/await vs Promises vs Observables.
**Answer:** Promises are one-time value resolution. Async/await makes Promise code readable. Observables handle multiple values over time (Streams).

### Q25: What's a memory leak and how do you prevent it?
**Answer:** When objects stay in memory after no longer needed. Prevent by: unsubscribing from observables, removing event listeners, nullifying references.

### Q26: How do you structure large CSS projects?
**Answer:** BEM (Block Element Modifier), SCSS variables for colors/spacing, mixins for patterns, separate files by component.

### Q27: Describe the component lifecycle in Angular.
**Answer:** ngOnInit (initialize), ngOnChanges (input changes), ngDoCheck (custom change detection), ngAfterViewInit (view ready), ngOnDestroy (cleanup).

### Q28: What's dependency injection and why does Angular use it?
**Answer:** DI provides dependencies to classes rather than creating them. Benefits: testability, flexibility, loose coupling, easier mocking.
